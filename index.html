SmartRoute — AI Token Optimizer

An intelligent dispatch layer that classifies every AI prompt and routes it to the cheapest capable model. 40–60% reduction in LLM API spend — zero quality loss. Read the launch post ↗

Problem

Developers use the most expensive model for every prompt. Haiku vs Opus is a ~50× cost gap. No tool solved the routing.

Approach

3-tier classification: classify → compress → route. Context compression cuts tokens 20–35% before routing runs. Feedback loop calibrates thresholds.

What's next

Replace heuristic classifier with a fine-tuned model trained on real prompt/quality pairs from the feedback loop.

When rendering virtualized lists of 10k+ items with stable keys, Million's block diffing traversed every node even when key order was unchanged — a wasted O(n) walk per render cycle. Engineering Principles

How technical decisions
move business needles.

Six beliefs I'll defend in any planning meeting — in terms of revenue, speed, and risk.

01 Revenue

Slow pages don't just annoy users. They kill conversion.

Google's own data: every 100ms of LCP costs ~1% in conversion rate. On a platform doing meaningful traffic, that's not a performance problem — it's a revenue leak with a measurable drip rate. I treat render budgets the same way a finance team treats burn rate: tracked weekly, justified explicitly, and never let slide quietly.

Moved Core Web Vitals 30% at Livemint — 1M+ users, every session faster
02 Speed

The right abstraction ships the tenth feature in hours, not days.

A component system that doesn't generalize compounds negatively: the fifth feature takes longer than the first. I design abstractions by mentally shipping the tenth feature first — if that thought experiment breaks the API, the abstraction is wrong before a line is written. This is the difference between a team that accelerates over time and one that slows under its own weight.

Widget system at Livemint cut layout deploy cycles from days to zero
03 Risk

Picking the wrong render strategy is an SEO liability, not a tech debt.

SSR, SSG, ISR, CSR — the choice looks like a technical preference until it costs you 60 points of search rank and three months of recovery time. I learned this expensively at LogiNext. Now I treat render strategy as a product decision: what's the content velocity, what's the SEO surface, what does the CDN cache invalidation actually look like in production? I ask this before the first PR is opened.

SEO up 60% at LogiNext after correcting the render strategy call
04 Cost

State that lives in the wrong layer generates support tickets, not just bugs.

When server state masquerades as UI state, you get stale data — users see the wrong thing, trust erodes, support tickets climb. When global state handles what should be local, you get unnecessary re-renders — perceived performance tanks, and nobody can pinpoint why. I define state boundaries before components, because the cost of getting this wrong isn't paid in code — it's paid in ops overhead and user churn.

Reduced API calls 30% at LogiNext by fixing state and caching layer boundaries
05 Speed

Cost of delay is real. Every week a feature ships late has a price tag.

Engineering bottlenecks on non-engineering decisions are the most expensive kind. When a marketer needs an engineer to reposition a banner, the cost isn't the hour of dev time — it's the campaign that launched late, the A/B test that didn't run, the opportunity that expired. I look for these loops and systematically eliminate them. Configuration over code. Autonomy over dependency. That's how engineering becomes a multiplier, not a gatekeeper.

No-code widget system gave editorial teams zero-latency deploy autonomy
06 Cost

Build vs Buy: the wrong call burns six months and a team's morale.

The instinct to build is strong in engineers — and sometimes right. But owning infrastructure you don't need to own is a silent tax on every sprint: maintenance cycles, incident response, onboarding overhead. I ask: is this core to our differentiation? If no — find the best library, contribute to it if needed, and spend the saved cycles on what actually moves the needle. I've done both, and I know where the line is.

Contributed to Next.js, TanStack Query, Million.js — use and improve, don't reinvent
